# Алгоритм роботи розробника з Git (Gitflow Basic)

Цей документ описує базовий алгоритм роботи з системою контролю версій Git, якого слід дотримуватися під час розробки проєкту. Основна мета - забезпечити стабільність основних гілок (`main` та `develop`) та організований процес внесення змін до кодової бази.

## Основні правила

1.  **Ніколи не комітьте безпосередньо в гілки `main` та `develop`.** Ці гілки повинні відображати стабільні версії коду.
2.  **Кожна нова фіча, виправлення помилки або експеримент розробляється у власній окремій гілці**, створеній на основі актуальної гілки `develop`.

## Покроковий алгоритм

1.  **Отримання актуальної версії коду:**

    * Переконайтеся, що ваша локальна гілка `develop` є актуальною:
        ```bash
        git checkout develop
        git pull origin develop
        ```

2.  **Створення власної гілки:**

    * Створіть нову гілку для вашої поточної задачі на основі гілки `develop`. Надайте гілці описову назву, що відображає її призначення (наприклад, `feature/new-login-form`, `bugfix/user-profile-error`, `experiment/alternative-api`).
        ```bash
        git checkout -b feature/your-feature-name
        ```

3.  **Написання коду:**

    * Внесіть необхідні зміни до кодової бази, реалізуючи функціональність, виправляючи помилку або проводячи експеримент.
    * Регулярно комітьте свої зміни. Пишіть чіткі та інформативні повідомлення до комітів, що описують внесені зміни.
        ```bash
        git add .
        git commit -m "feat: Додано нову форму логіну"
        ```
        або
        ```bash
        git add .
        git commit -m "fix: Виправлено помилку відображення профілю користувача"
        ```

4.  **Пуш змін у власну гілку:**

    * Після завершення роботи над задачею або досягнення проміжного етапу, запуште свою локальну гілку на віддалений репозиторій (origin).
        ```bash
        git push origin feature/your-feature-name
        ```

5.  **Створення запиту на злиття (Merge Request / Pull Request):**

    * Перейдіть на веб-інтерфейс GitLab (або іншу платформу, яку ви використовуєте).
    * Знайдіть свою віддалену гілку (`feature/your-feature-name`) та створіть запит на злиття (Merge Request) до гілки `develop`.
    * У описі Merge Request детально опишіть внесені зміни, їхню мету та будь-які важливі деталі для рецензентів.
    * Призначте рецензентів для перегляду вашого коду.

6.  **Перегляд коду (Code Review):**

    * Рецензенти переглянуть ваш код, залишать коментарі та можуть запросити внесення правок.
    * Внесіть необхідні зміни у свою локальну гілку та знову запуште їх на віддалений репозиторій. Ваш Merge Request автоматично оновиться.
    * Продовжуйте процес перегляду до тих пір, поки рецензенти не схвалять ваші зміни.

7.  **Злиття змін (Merging):**

    * Після схвалення вашого Merge Request, відповідальна особа (зазвичай тімлід або інший розробник з правами на злиття) зіллє вашу гілку (`feature/your-feature-name`) з гілкою `develop`.

8.  **Оновлення локальної гілки `develop`:**

    * Після успішного злиття, перейдіть на свою локальну гілку `develop` та оновіть її з віддаленого репозиторію.
        ```bash
        git checkout develop
        git pull origin develop
        ```

9.  **Перехід до наступної задачі:**

    * Для початку роботи над новою задачею, поверніться до кроку 2 та створіть нову гілку на основі оновленої гілки `develop`.

Дотримання цього алгоритму допоможе підтримувати чистоту та стабільність основних гілок, спростить процес перегляду коду та зробить розробку більш організованою та передбачуваною.
```
